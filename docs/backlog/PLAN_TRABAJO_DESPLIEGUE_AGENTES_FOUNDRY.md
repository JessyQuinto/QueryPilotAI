# Work Plan: Functional Deployment of Agents in Azure Foundry

## Objective
Design and implement an automated program in the project that:
- Creates and updates agents in Azure Foundry using versioned instructions.
- Configures agent memory (conversation memory and memory store where applicable).
- Generates output artifacts consumable by Infra and Backend.
- Enables reproducible deployments per environment (`dev`, `qa`, `prod`).

## Current Context
- The backend depends on these variables to operate:
  - `FoundryAgent__ProjectEndpoint`
  - `FoundryAgent__SqlPlannerAgentId`
  - `FoundryAgent__ResultInterpreterAgentId`
  - `FoundryAgent__ConciergeAgentId`
- There are Python scripts for agent creation, but currently they:
  - contain hardcoded values,
  - do not manage memory,
  - do not generate standardized outputs for `infra` and `backend`,
  - lack an idempotent flow per environment.

## Expected Result
An end-to-end deployment flow where a single command:
1. Validates prerequisites.
2. Creates/updates agents in Foundry.
3. Registers instructions per agent.
4. Configures memory and retention policies (if enabled in the environment).
5. Emits a configuration package for Bicep and Backend.
6. Allows post-deployment verification.

## Proposed Architecture of the Program

### Suggested Location
- `tools/foundry/deploy_agents.py`
- `tools/foundry/config/agents.<env>.yaml`
- `tools/foundry/templates/instructions/*.md`
- `tools/foundry/output/<env>/foundry-deployment.json`

### Declarative Input Configuration
File per environment: `tools/foundry/config/agents.<env>.yaml`

Minimum required fields:
- `environment`: `dev|qa|prod`
- `projectEndpoint`: Foundry project endpoint (not the Azure OpenAI endpoint)
- `modelDeployment`: Name of the model deployment
- `agents`: list of agents with:
  - `key`: `sqlPlanner|resultInterpreter|concierge`
  - `name`
  - `instructionsFile`
  - `temperature`, `top_p`, `max_tokens` (if applicable)
  - `memory.enabled`
  - `memory.storeName` (if applicable)
  - `tags`

### Mandatory Program Outputs
1. `tools/foundry/output/<env>/foundry-deployment.json`
2. `infra/parameters/foundry.<env>.json`
3. `backend/src/Functions.Api/appsettings.foundry.<env>.json`

Minimum output contract:
- `projectEndpoint`
- `agentIds` or version identifiers depending on the active SDK
- `agentNames`
- `agentVersions` (if applicable)
- `memoryStores`
- `generatedAtUtc`

## Integration with Infra and Backend

### Infra (Bicep/Parameters)
Add consumption of parameters generated by Foundry to publish application configuration:
- `FoundryAgent__ProjectEndpoint`
- `FoundryAgent__SqlPlannerAgentId`
- `FoundryAgent__ResultInterpreterAgentId`
- `FoundryAgent__ConciergeAgentId`

Recommendation:
- Do not inject secrets in plain text.
- Resolve secrets from Key Vault or secure references.

### Backend
Maintain the current environment variables contract in the Function App and add startup validations:
- Validate presence of endpoint and required agents.
- Fail fast with clear missing configuration messages.
- Structured load configuration version logging (without secrets).

## Phased Plan

## Phase 1 - Foundations and Security
Deliverables:
- Base structure `tools/foundry/`.
- Declarative configuration file per environment.
- Credential loading via `DefaultAzureCredential`.
- Elimination of hardcoded keys and endpoints.

Acceptance Criteria:
- The program fails if critical data is missing.
- No secrets are left exposed in the source code.

## Phase 2 - Agent Creation/Update
Deliverables:
- Idempotent `upsert` logic for agents.
- Reading instructions from versioned Markdown files.
- Stable mapping `key -> agentId/version`.

Acceptance Criteria:
- Running it twice does not duplicate agents.
- Changing instructions updates the target agent.

## Phase 3 - Memory and Conversational State
Deliverables:
- Persistent conversation configuration per agent.
- Support for managed memory store (if enabled in the Foundry environment).
- Safe fallback if the memory store is unavailable.

Acceptance Criteria:
- Agents maintain multi-turn context.
- Memory explicitly enabled or disabled per environment.

## Phase 4 - Artifacts Generation for Infra/Backend
Deliverables:
- Generation of `foundry-deployment.json`.
- Export parameters for `infra/parameters`.
- Export configuration for backend.

Acceptance Criteria:
- Infrastructure consumes parameters without manual editing.
- Backend starts with automatically generated configuration.

## Phase 5 - CI/CD Pipeline and Verification
Deliverables:
- Pipeline job to execute `deploy_agents.py` per environment.
- Post-deploy validations:
  - Existence of agents,
  - Simple invocation test,
  - Consistency with backend config.

Acceptance Criteria:
- Repeatable deployment from CI.
- Documented rollback on failure.

## Functional CLI Design (Program)

Suggested commands:
- `python tools/foundry/deploy_agents.py plan --env dev`
- `python tools/foundry/deploy_agents.py apply --env dev`
- `python tools/foundry/deploy_agents.py export --env dev`
- `python tools/foundry/deploy_agents.py verify --env dev`

Behavior:
- `plan`: Shows preview changes (create/update/no changes).
- `apply`: Executes changes and publishes agents.
- `export`: Generates artifacts for infra/backend.
- `verify`: Tests minimal invocation and validates final configuration.

## Agent Instructions Structure

Suggested files:
- `tools/foundry/templates/instructions/sql-planner.md`
- `tools/foundry/templates/instructions/result-interpreter.md`
- `tools/foundry/templates/instructions/concierge.md`
- `tools/foundry/templates/instructions/visualization-planner.md`

Current State:
- Official SQL Planner instruction defined in `tools/foundry/templates/instructions/sql-planner.md`.
- Official Result Interpreter instruction defined in `tools/foundry/templates/instructions/result-interpreter.md`.
- Official Visualization Planner instruction defined in `tools/foundry/templates/instructions/visualization-planner.md`.
- Official Concierge instruction defined in `tools/foundry/templates/instructions/concierge.md`.
- Initial environment configuration created in `tools/foundry/config/agents.dev.yaml`.
- Base automation helper created in `tools/foundry/deploy_agents.py` with `plan` and `export` commands.
- The deployment program should read this file and publish it to the agent mapped by the key `sqlPlanner`.
- The `FoundryAgent__SqlPlannerAgentId` variable should store the deployed agent's identifier/version, not the instructions text.

Best Practices:
- Instructions per single responsibility.
- Structured JSON output when applicable.
- Prompt versioning with brief changelog.
- Regression tests per instruction.

## Risks and Mitigations
- Foundry SDK changes (IDs vs agent name/version):
  - Mitigate with an adapter layer and stable output contract.
- Capability differences per region/model:
  - Pre-validation of model and region in `plan`.
- RBAC permission failure:
  - Permissions precheck at the start of `apply`.
- Drift between deployed agents and backend config:
  - Mandatory `verify` and configuration hash.

## Definition of Done
Considered complete when:
- The program deploys agents per environment without hardcoding,
- Generates configuration usable by infra and backend,
- Backend can start and respond utilizing newly deployed agents,
- Automated post-deployment verification exists,
- Operations and rollback are documented.

## External Dependencies
- Azure AI Foundry Project enabled.
- Model deployed and available in the target region.
- Identity with permissions to manage agents.
- Azure Key Vault for secrets.
